OHCredits — cloud “event” workflow (free Supabase tier)
=======================================================

Why: Static GitHub Pages cannot run a private write API. Supabase gives you a real
database + HTTPS API. The player only needs the public “anon” key and reads rows
by event code. Publishing uses a secret that never ships in the player page.

This repo is wired for project ref: uyufnbroqmwjtzvcsosv
  API URL: https://uyufnbroqmwjtzvcsosv.supabase.co
  (see player-config.json — you must paste the anon key from Dashboard → Settings → API)

Security
--------
If your database password was ever pasted into chat or committed to git, open
Supabase → Project Settings → Database → reset the database password immediately.
Never commit connection strings or passwords to this repository.

1) Create a project at https://supabase.com (free tier is enough for many shows).

2) Apply the database migration (pick one):

   A) Easiest: Dashboard → SQL → New query → paste the full contents of:
      supabase/migrations/20250328000000_credit_events.sql
      → Run.

   B) CLI: install/use psql, or from repo root in PowerShell:
      $env:DATABASE_URL = "<paste URI from Dashboard → Database; use pooler/Session if needed>"
      .\scripts\apply-migration.ps1

   Note: Host db.<ref>.supabase.co is often IPv6-only. If tools fail to connect,
   use the “Connection pooling” / Session mode URI from the dashboard (usually IPv4).

3) Install Supabase CLI (optional) and deploy the Edge Function, or paste
   supabase/functions/publish-credits/index.ts into the Dashboard → Edge Functions
   → create “publish-credits”.

   Set project secret: PUBLISH_SECRET = a long random string (password manager).

   IMPORTANT — browser publish from GitHub Pages (CORS + JWT gateway):
   Supabase can validate JWT *before* your function runs. A browser OPTIONS
   preflight has no valid JWT, so you usually need JWT verification OFF for
   publish-credits, and you must redeploy after changing it.

   • CLI (recommended): log in once (npx supabase login), then from repo root:
       PowerShell:  .\scripts\deploy-publish-credits.ps1
       Or manually: npx supabase functions deploy publish-credits ^
         --project-ref uyufnbroqmwjtzvcsosv --no-verify-jwt --use-api --yes
     Use --use-api on Windows to avoid Docker. CI can set SUPABASE_ACCESS_TOKEN
     instead of interactive login (Dashboard → Account → Access Tokens).

   Troubleshooting — “Requested function was not found” (HTTP 404):
     The function is not deployed to THIS project ref. Common causes:
     - Deployed from the Dashboard into a different Supabase project than the
       one wired in app.js (DEFAULT_PUBLISH_FUNCTION_URL) / player-config.json.
     - Function name is not exactly publish-credits (hyphen, not underscore).
     - Old deploy: open Edge Functions in the Dashboard and confirm
       publish-credits exists for project uyufnbroqmwjtzvcsosv, then redeploy.
     Quick check (expect 401 JSON body, not 404):
       curl -X POST "https://uyufnbroqmwjtzvcsosv.supabase.co/functions/v1/publish-credits" ^
         -H "Content-Type: application/json" -d "{}"

     This repo’s supabase/config.toml sets verify_jwt = false for
     publish-credits when you deploy with the CLI against a linked project.

   • Dashboard-only: Edge Functions → publish-credits → Settings → turn OFF
     JWT verification / “Verify JWT” for this function, then redeploy or save
     so the change applies.

   The editor also loads player-config.json and, when supabaseAnonKey is set,
   sends Authorization: Bearer <that key> plus X-OHCredits-Publish-Secret for
   your PUBLISH_SECRET. That helps when JWT verification stays ON and the key
   is accepted as a gateway token—but preflight can still fail until JWT is off
   for this function; if you use the new sb_publishable_* keys, keep verify_jwt
   off unless Supabase documents otherwise for your project.

   The editor posts to a fixed Edge Function URL: `DEFAULT_PUBLISH_FUNCTION_URL`
   in `app.js` (forks: change it there). Correct shape:
   https://YOUR_REF.supabase.co/functions/v1/publish-credits

4) On your GitHub Pages site (same folder as index.html / player.html):

   - player-config.json — copy player-config.example.json and fill in:
     - supabaseUrl: Project Settings → API → Project URL
     - supabaseAnonKey: Project Settings → API → anon public key
     Commit player-config.json. The anon key is safe here if RLS only allows SELECT
     on credit_events (see migration). Never commit the service_role key.

5) In the OHCredits editor → “Cloud events”:
   - Event code: e.g. OfficeHours
   - Publish secret: same as PUBLISH_SECRET (typed in the page; stored only in
     browser localStorage, not in JSON)
   Click “Publish design”, then copy “16:9 / 9:16 URL (?event)” for third parties.

6) Third parties open:
   https://YOUR_SITE/.../player.html?view=16x9&event=officehours
   (or 9:16). Their browser loads player-config.json from your site, then fetches
   the design from Supabase. No giant hash in the URL.

   vMix / fixed canvas: &hd=1 or &hd=1080 → 1920×1080 (or 1080×1920 vertical).
   1440p: &hd=1440 or &hd=1440p or &hd=qhd (or &output=1440 / &output=qhd / &output=2560)
   → 2560×1440 or 1440×2560 vertical.
   4K: &hd=4k (or &hd=2160, &scale=2, &scale=200) → 3840×2160 or 2160×3840 —
   match the browser input resolution and use 100% zoom so the page isn’t
   upscaled by the app (scaling stays sharp).

Alternatives: Firebase Firestore, Cloudflare D1 + Worker, PlanetScale + serverless
API — same pattern: public read by event id, writes behind a secret or auth.
